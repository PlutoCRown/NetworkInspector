# BACKGROUND 
这个项目主要是为了实现一个网络检查器，它的主要核心包是一个 Chrome extension，用户安装了这个插件之后，它可以创建一个规则组，它包含四个部分，第一个部分是它会匹配哪些网站，在哪些网站上生效。第二个规则是它会捕获哪些网络请求？第三个规则是它捕获了网络请求之后，它从哪里去提取我们的结构化数据？第四个是它提取完结构化数据之后，如何在我们的网络面板上渲染在我们的插件的视图上去渲染一个规则组，就包含这四个部分，每个部分其实还可以区分普通模式和高级模式。
普通模式的话我们会提供一个。表单式的创建选项去方便他创建一些比较预设式的 UI 高级模式的话，允许用户去自定义创建一些，比如说他们的网络请求是有 Protobuf 或者什么加密的，这种普通模式就没有办法解析中间的数据，他们就可以通过高级模式去。创建一些自定义的解码流程，可以创建直接插入一些 JS 的执行代码去做这个数据的解析，因为这个东西它是面向于普通用户去用的，所以它的主要界面还是会引导用户，你一进来我给你提供一个文档站，一个 Playground 的教程，一个 create 的入口，还有一个 import 的。导入口，它可以允许技术人员进行创建，之后将一个配置文件发送给普通人员，他导入这个配置文件，立刻就可以开始在一个网站上进行埋点的捕获。

# 技术
这是一个基于 bun 的 Mona repo 项目。
目前只初始化第一个子包，也就是 Chrome 的 extension。
第一个是，它的工具栏要允许打开侧边栏。侧边栏是主要的信息内容渲染的地方，工具栏还可以快捷地开启或者关闭这个规则组，也可以引导用户直接打开一个新窗口，在里面编辑和创建规则组。
第二个是，整个项目需要使用 shadcn 的组件库来做 UI，然后可以找一个比较好用的 Chrome extension template。
然后我们来详细定义设置页面里的规则组。一个规则组包含哪些内容，需要明确下来。

1. 第一个部分是匹配哪些网站，这里是一个正则表达式列表。

2. 第二个部分是它会捕获哪些网络请求，这里也是一个正则表达式列表。

3. 然后是捕获之后，怎么去结构化这些数据。需要注意的是，上面捕获到的网络请求 URL，实际上对于每种匹配规则来说，能够匹配到的网络请求都不一样，所以针对内容的信息提取，也应该为每一个 URL 提供一个单独的编辑板块。

4. 在这个编辑板块里，可以选择如何去读取请求中的信息。下面是一行一行的预设项，比如 title、attributes 这些字段。用户可以在输入框里快捷输入一些选项，比如可以从 query 参数中取，可以从 JSON body 里面取，也可以从 form-data 里面取, 可以从 request header 里面取。

当用户输入这些之后，就相当于有一个下拉框，让他选中这几个 tag。选中完之后，后面还可以继续输入它们的 attribute。key path. 这里的逻辑有点像 GitHub 的搜索框：前面的一个 tag 用来表明是在当前仓库搜索，或者在当前组织搜索，后面才是具体的搜索内容。类似于这样的输入框。

5. 这个编辑板块的前面要先选择这个 Renderer。

我们预设几种 Renderer。

1. 第一种 Renderer 支持 title: string popover: object 这样的结构，那么这个编辑板块下面，其实只包含两个输入框，允许用户输入。
2. 第二种 Renderer 支持 title: string desc: string expend: object 这样的结构，那么这个编辑板块下面，其实只包含三个输入框，允许用户输入。

最后，我们在侧边栏这里显示当前捕获组捕获到的所有内容。
然后，它根据所选的 Render 来渲染当前的卡片。
如果用户选择了自定义，那么他可以自己编写一个类似 HTML templates 这样的东西来实现渲染。

最后，配置项还支持设置 highlights 和 alias 和 filter，可以配置其生效范围（在renderer的哪个板块）
alias 允许在捕获到匹配的字符串内容时，把它替换成你想要的名字。
highlights 允许在捕获到特定内容时进行高亮。高亮的渲染样式由 Renderer 决定。
Filter 决定在某一项匹配到特定内容的时候就把它过滤掉，或者说把 Json 中的内容过滤掉。
